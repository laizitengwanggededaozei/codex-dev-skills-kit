# 开发交流规范

本规范用于降低复杂任务中的沟通成本，同时保持编码闸门足够紧。详细证据、
方案和审查仍写入 `docs/ai-dev`，这里仅定义对话和文档开头的交流格式。

## 1. 交流状态块

复杂任务、长程任务、预编码审查、编码后审查和最终报告开头使用短状态块：

```text
当前阶段：需求对齐 / 预编码审查 / 编码中 / 编码后审查 / 待验收
编码闸门：关闭 / 已打开：开始编码 / 已打开：自主执行
本轮目标：...
已确认：...
待确认：最多 1 个当前必须回答的问题
可后置确认：不影响当前阶段推进的问题
下一步：...
```

规则：

- 状态块只放关键状态，不替代正文证据。
- `待确认` 默认最多 1 个当前必须回答的问题。
- 不急着回复的问题放入 `可后置确认`。
- 术语优先沿用现有文档习惯：`待确认`、`未确认项`、`编码闸门`、
  `编码授权`、`下一步`、`验证缺口`、`剩余风险`、`目标回归`、`审查结论`。
- 不使用“阻断确认”作为栏目名；`阻断` 只用于真实风险或业务行为，例如
  阻断编码、阻断提交、阻断验收。

## 2. 未确认项澄清

用户回答待确认问题时，经常是在补充实际情况、澄清需求或纠正 Codex 理解，
不一定是在授权继续。

出现以下情况时进入“未确认项澄清”：

- Codex 的理解混乱、假设过多或前后不一致。
- 用户补充新的业务背景或实际情况。
- 用户纠正目标、范围、方案或授权状态。
- 用户表达“进入某方案、确认方向、继续完善”，但没有明确要求修改文件。

处理方式：

- 先复述更新后的理解。
- 把用户补充分成 `已确认`、`修正后的需求`、`仍待确认`。
- 如果仍需用户回复，只保留 1 个当前必须回答的问题。
- 其他问题放入 `可后置确认`。
- 未出现 `开始编码` 或 `自主执行` 时，编码闸门保持关闭。

## 3. 编码闸门

编码闸门默认关闭。只有下面两类明确表达可以打开：

1. `开始编码`
2. `自主执行`

等价表达必须同时包含“开始/执行/修改/落地”等动作授权，例如：

- `开始修改文件`
- `按方案 C 开始改`
- `按计划落地实现`

以下表达一律不打开编码闸门：

- `进入 C 方案`
- `方案 C`
- `按 C`
- `确认方案`
- `方案可以`
- `继续`
- `下一步`
- `完善一下`
- `同意这个方向`
- `先这样`

这些只代表继续对齐、完善、审查或压缩待确认问题。

## 4. 快速提示词

复杂任务预编码：

```text
请按复杂任务标准流程处理。先不要编码。
先进行任务理解、skill 使用计划、计划审查、需求/方案对齐，并沉淀计划和审查文档。
每轮开头给出交流状态块；待确认最多 1 个，其余放可后置确认。
只有我明确说“开始编码”或“自主执行”时，才允许修改正式文件。
```

澄清实际情况：

```text
先进入未确认项澄清。
请复述你更新后的理解，把我的补充分成已确认、修正后的需求、仍待确认。
不要编码；如果还需要我回答，只问 1 个当前必须回答的问题。
```

编码后审查：

```text
请做编码后目标回归和 diff 风险审查。
先判断是否满足原始目标，再列风险、验证缺口、剩余风险。
最终报告开头保留交流状态块。
```

## 5. 剩余待办

- 当前目标必须处理的问题，留在当前任务和编码后审查中。
- 不影响当前验收的 P2/P3 优化，写入 `剩余风险` 或 `可后置确认`。
- 长程任务中需要后续追踪的事项，沉淀到 `docs/ai-dev/60_交接记录/`。
