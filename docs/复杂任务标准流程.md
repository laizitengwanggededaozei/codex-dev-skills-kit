# 复杂任务标准流程

这份流程用于迁移到新机器或新项目后，统一 Codex 处理复杂任务的方式。
它不是所有小任务都强制执行；但一旦用户显式触发，或 Codex 判断任务
足够复杂，就必须完整执行。

开发交流格式见 `docs/开发交流规范.md`。本流程只保留必须遵守的入口和闸门。

## 1. 什么时候触发

用户显式触发：

- "复杂任务"
- "长程任务"
- "标准流程"
- "完整流程"
- "先计划"
- "先审查"
- "先对齐"
- "编码前确认"
- "先使用 skill 计划"

Codex 自动触发：

- 从零到一的项目、产品、架构设计
- 多仓库任务
- 后端加前端或 App 联动
- 多页面、多 bug、多个症状同时定位
- 权限、数据可见性、报表统计口径、分支过滤、SQL、迁移
- 业务边界、系统归属、接口契约不清楚

## 2. 复杂度分级

### 极其复杂

适用：从零到一项目、大型架构、平台迁移。

要求：至少 3 轮：

1. 任务理解
2. skill 使用计划
3. 计划审查
4. 需求/方案对齐

每轮结束后，Codex 必须提示下一轮应补充的计划方向。除非用户确认进入
编码，否则不开始写代码。

### 复杂

适用：前后端加 App、多页面、多 bug、报表口径和分支过滤联动。

要求：至少 1 轮：

1. 任务理解
2. skill 使用计划
3. 计划审查
4. 需求/方案对齐

一轮后 Codex 给出分支：进入下一轮，或开始编码。若 Codex 判断范围仍
然复杂，应主动询问是否进入第二轮。

### 非简单任务

适用：单功能但边界不清、单 bug 但 owner layer 不明、共享 API 修改。

要求：1 轮轻量对齐后再编码。

## 3. skill 搭配和时机

| 阶段 | skill | 用途 |
|---|---|---|
| 需求理解 | `codex-question-aligner` | 整理意图、事实、假设、问题和下一句提示 |
| 计划/边界对齐 | `codex-design-alignment` | 对齐业务边界、系统归属、skill 触发时机、方案和风险 |
| bug 根因定位 | `codex-diagnose` | 先列可验证假设，再定位 owner layer |
| 工程执行 | `engineering-ai-dev-suite` | 方案确认后编码，复用项目模式 |
| 编码后审查 | `codex-code-review` | 审查 diff、API/data 风险、缺失验证 |
| 项目初始化 | `setup-codex-dev-workspace` | 检查或创建 `docs/ai-dev` 结构和命名规则 |

`codex-diagnose` 可以和 `codex-design-alignment` 组合使用：当 bug 根因
和业务口径、权限边界、分支可见性、系统归属有关时，不能只做技术定位。

编码结束后，`codex-code-review` 必须和目标回归审查组合使用：先判断
是否满足原始任务目标，再看 diff 风险。

## 4. 编码前必须沉淀什么

触发复杂任务流程后，编码前必须有两份文档：

- 计划/设计文档：`docs/ai-dev/20_设计方案/ADR-YYYYMMDD-NNN-title.md`
- 审查文档：`docs/ai-dev/30_审查记录/REVIEW-YYYYMMDD-NNN-scope.md`

复杂任务每轮计划、审查和最终报告开头必须有交流状态块：

```text
当前阶段：...
编码闸门：关闭 / 已打开：开始编码 / 已打开：自主执行
本轮目标：...
已确认：...
待确认：最多 1 个当前必须回答的问题
可后置确认：不影响当前阶段推进的问题
下一步：...
```

设计文档和审查文档默认必须使用中文，除非用户明确要求使用其他语言。

计划/设计文档至少写：

- 目标
- 已确认事实
- 待确认问题
- 任务复杂度和触发原因
- 用户目标和非目标
- 涉及仓库、模块、页面、接口
- 每个阶段使用哪个 skill
- 当前是第几轮计划，结论是什么
- 业务边界、数据/API 契约、权限或口径
- 风险
- 建议方案
- 下一步
- 需要沉淀的文档
- 实施计划
- 验证计划

计划/设计文档必须契合 `codex-design-alignment` 的要求：围绕目标、已确认、
待确认、风险、建议方案、下一步和需要沉淀的文档展开；不要把它写成泛泛
开发记录。

审查文档至少写：

- findings first，先列问题，不先写总结
- 严重级别，例如 P0/P1/P2/P3
- 文件和行号，能定位就必须定位
- 影响
- 证据
- 建议
- 计划是否覆盖用户目标
- 哪些假设仍未确认
- 验证缺口
- 目标回归覆盖情况
- 是否允许进入编码
- 需要用户确认什么

审查文档必须契合 `codex-code-review` 的要求：问题优先、按严重级别排序、
有影响、有证据、有建议，并说明验证缺口和目标回归结果。

## 5. 编码闸门

默认规则：编码闸门关闭。计划/设计文档和审查文档完成后，也不能自动编码。

只有下面两类明确表达可以打开编码闸门：

1. `开始编码`
2. `自主执行`

等价表达必须同时包含“开始/执行/修改/落地”等动作授权，例如 `开始修改文件`、
`按方案 C 开始改`、`按计划落地实现`。

以下表达一律不打开编码闸门：`进入 C 方案`、`确认方案`、`方案可以`、`继续`、
`下一步`、`完善一下`、`同意这个方向`、`先这样`。这些只代表继续对齐、
完善、审查或压缩待确认问题。

用户回答待确认问题时，如果是在补充实际情况、纠正理解或重新定义需求，
Codex 必须先进入未确认项澄清：复述更新后的理解，拆分为已确认、修正后的
需求、仍待确认。未出现 `开始编码` 或 `自主执行` 时，编码闸门保持关闭。

如果代码已经改了一部分，必须暂停并做回放式审查。只有用户明确要求跳过，
才可以不做回放式审查。

## 6. 编码后必须做什么

1. 目标回归审查：只对照原始任务目标和要求，不扩大成泛泛 review。
2. diff 风险审查：使用 `codex-code-review` 看接口、权限、数据、分页、
   统计口径、缺失验证、误改范围。
3. 仓库验证：按项目 `AGENTS.md` 跑对应仓库的 `git diff --check`、lint、
   compile 或已有测试。
4. 最终报告：说明目标覆盖、修改文件、验证结果、剩余风险、文档链接。

## 7. 推荐提示词

```text
请按复杂任务标准流程处理。先不要编码。
先进行任务理解、skill 使用计划、计划审查、需求/方案对齐，并在
docs/ai-dev/20_设计方案 和 docs/ai-dev/30_审查记录 下沉淀计划和审查文档。
每轮开头给出交流状态块；待确认最多 1 个，其余放可后置确认。
只有我明确说“开始编码”或“自主执行”时，才允许修改正式文件。
```

如果已经发生过代码修改：

```text
请按复杂任务标准流程做回放式审查。
先审查当前未提交 diff 是否覆盖原始目标、是否存在风险、是否需要补计划。
把审查结果沉淀到 docs/ai-dev/30_审查记录，完成后等待我确认。
```
