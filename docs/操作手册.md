# Codex Dev Skills Kit 操作手册

这本手册只讲安装完成以后怎么用。你不需要记脚本，不需要理解目录实现，只要按场景复制提示词发给 Codex。

## 1. 你现在有什么能力

安装完成后，Codex 有这些常用能力：

| 能力 | 用途 | 你怎么点名 |
|---|---|---|
| Engineering AI Dev Suite | 通用开发、重构、迁移、文档、兜底处理 | `使用 Engineering AI Dev Suite，...` |
| Setup Codex Dev Workspace | 初始化项目的 AI 开发目录和规则 | `使用 Setup Codex Dev Workspace，...` |
| Codex Question Aligner | 把一段模糊想法整理成精确问题 | `使用 Codex Question Aligner，...` |
| Codex Design Alignment | 需求、方案、架构、迁移前对齐 | `使用 Codex Design Alignment，...` |
| Codex Code Review | 审查代码、未提交修改、PR、SQL、配置 | `使用 Codex Code Review，...` |
| Codex Diagnose | 诊断 bug、测试失败、接口异常、错误数据 | `使用 Codex Diagnose，...` |

你不用一次记住所有名字。日常最常用的是：

```text
使用 Codex Question Aligner，...
使用 Codex Design Alignment，...
使用 Codex Code Review，...
使用 Codex Diagnose，...
```

## 2. 第一次进入一个项目怎么说

第一次进入项目，不要马上让 Codex 改代码。先让它认识项目。

复制这句：

```text
使用 Setup Codex Dev Workspace，初始化当前项目。
先检查当前目录是单仓库还是多仓库工作区，再告诉我你准备创建哪些目录和文件。
请区分“结构初始化完成”和“编码上下文初始化完成”。
如果可以确认项目事实，请填充 docs/ai-dev/01_PROJECT_CONTEXT.md；
如果不能确认，请标记 TBD、待确认或可后置确认。
不要覆盖已有文件。
```

Codex 应该做这些事：

- 看当前目录。
- 判断是单仓库，还是包含多个仓库的工作区。
- 检查有没有 `AGENTS.md`。
- 检查有没有 `docs/ai-dev/`。
- 检查有没有 `scripts/ai-dev/`。
- 检查项目构建和测试方式。
- 检查源码入口、模块边界、文档、脚本、配置和外部系统线索。
- 先给初始化计划。
- 再创建缺失内容。
- 把已确认项目事实写入项目文档，把未知项标记出来。

如果运行环境支持并且你要求它使用子代理，Codex 可以让子代理只读探索
目录、文档、脚本、配置和源码入口。子代理只是加速探索；最终由主代理
整合事实、处理冲突并写入文档。没有子代理时，Codex 也应该按同一清单
自己探索。

你可以简单理解：

- `AGENTS.md`：入口摘要、硬规则和详细文档链接。
- `docs/ai-dev/01_PROJECT_CONTEXT.md`：项目目的、仓库地图、入口、命令、验证和风险。
- `docs/ai-dev/02_WORKFLOW_RULES.md`：项目级流程、验证顺序和审查重点。
- `docs/ai-dev/60_交接记录/`：探索摘要、证据、冲突、未知项和交接总结。
- `docs/ai-dev/`：长期记录需求、设计、审查、bug、执行手册、交接总结。
- `scripts/ai-dev/`：放以后可以重复执行的项目脚本。

## 3. 你说不清楚时怎么问

当你脑子里有一段话，但不知道怎么问，先不要直接让 Codex 动手。

复制这句：

```text
使用 Codex Question Aligner，帮我把下面这段话整理成精确问题、我的真实诉求、需要确认的点和下一句应该怎么问：
【粘贴你的原话】
```

Codex 应该输出：

- 你的核心意思。
- 这段话里有哪些事实。
- 这段话里有哪些假设。
- 你真正要确认的问题。
- 哪些可以查代码或文档。
- 哪些需要你或团队拍板。
- 下一句最适合发给 Codex 的提示词。

例子：

```text
使用 Codex Question Aligner，帮我把下面这段话整理成精确问题、我的真实诉求、需要确认的点和下一句应该怎么问：
我感觉这个接口和前端页面对不上，但是我也不知道是不是分页的问题，测试也没提，我怕删错。
```

## 4. 做方案前怎么对齐

当你要做一个新功能、迁移、重构、接口调整、权限调整，不要直接开始写。

复制这句：

```text
使用 Codex Design Alignment，帮我对齐这个方案。
先查现有代码和文档，说明已确认事实、待确认问题、影响范围、风险和推荐下一步。
方案如下：
【粘贴你的方案】
```

Codex 应该重点看：

- 业务目标是什么。
- 涉及哪些端或哪些模块。
- 谁拥有这个逻辑。
- 接口、数据、权限、路由是否受影响。
- 是否需要兼容旧数据或旧页面。
- 是否需要写 ADR 或设计文档。

## 4.1 复杂任务标准流程

当任务是长程任务、复杂任务、多仓库任务、多页面或多 bug 修复时，不要
直接进入编码。复制这句：

```text
请按复杂任务标准流程处理。先不要编码。
先进行任务理解、skill 使用计划、计划审查、需求/方案对齐，并在
docs/ai-dev/20_设计方案 和 docs/ai-dev/30_审查记录 下沉淀计划和审查文档。
每轮开头给出交流状态块；待确认最多 1 个，其余放可后置确认。
只有我明确说“开始编码”或“自主执行”时，才允许修改正式文件。
```

Codex 也应该在判断任务复杂时自动触发这个流程。典型场景：

- 从零到一项目
- 前后端加 App 联动
- 多页面、多 bug、多个症状一起修
- 权限、数据可见性、报表统计口径、分支过滤、SQL 或迁移

复杂任务会按阶段使用合适的 skill：

- `codex-question-aligner`：整理意图和问题
- `codex-design-alignment`：对齐业务边界、系统归属、skill 触发时机和方案
- `codex-diagnose`：定位 bug 根因
- `engineering-ai-dev-suite`：方案确认后编码
- `codex-code-review`：编码后审查 diff 和风险

默认规则：编码闸门关闭。计划和审查文档完成后，Codex 也不能自动编码。
只有你明确写出 "开始编码" 或 "自主执行" 时，Codex 才能修改正式文件。
"确认方案"、"进入方案 C"、"继续"、"下一步"、"完善一下" 都只代表继续对齐。

计划/设计文档和审查文档默认必须写中文，除非你明确要求其他语言。

- 计划/设计文档要贴合 `codex-design-alignment`：写清目标、已确认事实、
  待确认问题、影响范围、数据/API/权限影响、风险、建议方案、下一步和
  需要沉淀的文档。
- 审查文档要贴合 `codex-code-review`：问题优先，按 P0/P1/P2/P3 排序，
  尽量给文件和行号，并写清影响、证据、建议、验证缺口和目标回归结果。

如果代码已经改了一部分，复制这句：

```text
请按复杂任务标准流程做回放式审查。
先审查当前未提交 diff 是否覆盖原始目标、是否存在风险、是否需要补计划。
把审查结果沉淀到 docs/ai-dev/30_审查记录，完成后等待我确认。
```

完整规则见：

```text
docs/复杂任务标准流程.md
```

开发交流格式看：

```text
docs/开发交流规范.md
```

## 4.2 开发交流怎么说

当你觉得 Codex 理解混乱、待确认问题太多，或者你需要补充实际情况时，复制这句：

```text
先进入未确认项澄清。
请复述你更新后的理解，把我的补充分成已确认、修正后的需求、仍待确认。
不要编码；如果还需要我回答，只问 1 个当前必须回答的问题。
```

当你要让 Codex 压缩待确认问题时，复制这句：

```text
请把待确认压缩到 1 个当前必须回答的问题。
其他问题放到可后置确认，不要把方案确认理解为编码授权。
```

当你确认可以修改正式文件时，只需要明确写：

```text
开始编码
```

或者：

```text
自主执行
```

## 5. 每天开发怎么说

复制这个模板：

```text
使用 Engineering AI Dev Suite，实现【功能名称】。
要求：
1. 先看现有代码怎么写
2. 复用项目已有风格
3. 不做无关重构
4. 改完运行能找到的测试
5. 最后告诉我改了哪些文件、验证了什么
```

例子：

```text
使用 Engineering AI Dev Suite，实现订单列表按状态筛选。
要求：
1. 先看现有列表筛选怎么写
2. 前后端参数名保持一致
3. 不做无关重构
4. 改完运行能找到的测试
5. 最后告诉我改了哪些文件、验证了什么
```

## 6. 修 bug 怎么说

复制这个模板：

```text
使用 Codex Diagnose，诊断【问题描述】。
先不要改代码。
先给我 3 到 5 个可能原因，每个原因说明怎么验证。
确认根因后再修。
```

例子：

```text
使用 Codex Diagnose，诊断登录成功后跳转空白页。
先不要改代码。
先给我 3 到 5 个可能原因，每个原因说明怎么验证。
确认根因后再修。
```

修完以后补一句：

```text
把这次 bug 的现象、根因、修复、验证沉淀到 docs/ai-dev/40_故障修复/，文件名按项目规则生成。
```

## 7. review 怎么说

复制这个模板：

```text
使用 Codex Code Review，review 当前未提交修改。
只看真正可能出问题的地方。
重点看：
1. 接口是否不兼容
2. 权限或路由是否有问题
3. 分页、筛选、统计口径是否错误
4. 是否漏了测试
5. 是否有死代码或重复 API
不要输出无关格式建议。
```

好的 review 应该像这样：

```text
P1 src/api/order.js:42 - 分页参数名和后端不一致
影响：点击第二页仍然返回第一页数据。
证据：前端传 pageNo，后端读取 currentPage。
建议：统一参数名，或者在请求构造层做兼容。
```

如果你只想审查，不想修，明确说：

```text
只审查，不要修改文件。
```

如果你想审查后直接修，明确说：

```text
先输出审查发现，等我确认后再修。
```

## 8. 多仓库怎么用

如果一套项目有后端、前端、App，不要所有事情都在一个 Codex 窗口里做。

推荐这样分：

| 窗口 | 打开位置 | 做什么 |
|---|---|---|
| 总工作区窗口 | 总项目目录 | 跨端方案、总体计划、总结 |
| 后端窗口 | 后端仓库目录 | 接口、数据库、服务逻辑 |
| 前端窗口 | 前端仓库目录 | 页面、路由、组件、接口调用 |
| App 窗口 | App 仓库目录 | App 页面、移动端接口、端上逻辑 |

第一次进入总工作区时说：

```text
使用 Setup Codex Dev Workspace，初始化当前多仓库工作区。
请识别这里有哪些仓库，并生成仓库地图。
```

第一次进入某个具体仓库时说：

```text
使用 Setup Codex Dev Workspace，初始化当前仓库。
请只记录这个仓库自己的构建、测试、目录和开发规则。
```

跨端总结放总工作区；单端实现细节放对应仓库。

## 9. 文档放哪里

初始化后，文档统一放在：

```text
docs/ai-dev/
```

常见位置：

| 记录内容 | 放哪里 |
|---|---|
| 需求说明 | `docs/ai-dev/10_需求记录/` |
| 设计方案 | `docs/ai-dev/20_设计方案/` |
| 代码审查 | `docs/ai-dev/30_审查记录/` |
| bug 修复记录 | `docs/ai-dev/40_故障修复/` |
| 测试服/生产执行步骤 | `docs/ai-dev/50_执行手册/` |
| 阶段交接总结 | `docs/ai-dev/60_交接记录/` |
| 临时排查记录 | `docs/ai-dev/90_临时笔记/` |

你可以直接说：

```text
把这次修复沉淀到 docs/ai-dev/40_故障修复/，文件名按项目规则生成。
```

或者：

```text
把这次测试服务器执行步骤沉淀到 docs/ai-dev/50_执行手册/，写成可以照着执行的版本。
```

## 10. 文件命名怎么看

文件名前面的英文和数字是为了好找。

常见命名：

```text
REQ-20260515-001-login-session.md
ADR-20260515-001-api-pagination-contract.md
REVIEW-20260515-001-uncommitted-diff.md
BUGFIX-20260515-001-login-blank-page.md
RUNBOOK-20260515-001-test-db-menu-repair.md
HANDOFF-20260515-001-frontend-backend-split.md
TMP-20260515-001-debug-token-refresh.md
```

你不用自己想文件名，可以说：

```text
按 docs/ai-dev 的命名规则帮我创建一份 bug 修复记录。
```

## 11. 一到两个月项目怎么保持不乱

每天只遵守这几条：

1. 重要需求放 `10_需求记录/`。
2. 重要设计放 `20_设计方案/`。
3. 每次大 review 放 `30_审查记录/`。
4. 每个重要 bug 放 `40_故障修复/`。
5. 需要别人照着执行的步骤放 `50_执行手册/`。
6. 一周或一个阶段结束，放一份 `60_交接记录/`。
7. 临时排查先放 `90_临时笔记/`，有价值再整理。

每周可以让 Codex 整理一次：

```text
使用 Engineering AI Dev Suite，整理本周 docs/ai-dev 下的记录。
请告诉我：
1. 哪些临时记录应该归档
2. 哪些 bugfix 需要补验证
3. 哪些 runbook 可以复用
4. 生成一份本周 handoff 总结
```

## 12. Codex 没按预期做怎么办

直接这样说：

```text
使用 Engineering AI Dev Suite，刚才你的执行不符合我的预期。
问题是：【说清楚哪里不对】。
请先总结偏差原因，再告诉我应该怎么修改 skill 或项目规则。
不要先改代码。
```

例子：

```text
使用 Engineering AI Dev Suite，刚才你的 review 不符合我的预期。
问题是：你先总结优点，没有先输出风险。
请先总结偏差原因，再告诉我应该怎么修改 skill 或项目规则。
不要先改代码。
```

如果确认要调优 skill，再说：

```text
使用 Engineering AI Dev Suite，调优 engineering-ai-dev-suite。
请修改 code review 规则，强化 findings first，并增加一个正确输出示例。
```
